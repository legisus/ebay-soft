# Governance — open source, license, donations, growth

How we structure ebay-soft as a project: what's public, what's paid, what licenses apply, and how we get from "solo founder" to "real company" without giving away the moat.

---

## TL;DR

- **Application code = source-available, not open-source.** Use Business Source License 1.1 (BSL) with a 4-year clock to Apache 2.0. Customers can read every line; a competitor can't run a hosted copy.
- **Selected artifacts = real open source (Apache 2.0).** Java + TypeScript eBay SDKs, the listing-title scorer, the image-background-remover wrapper, the free calculators. These drive top-of-funnel and trust.
- **Revenue = subscriptions, not donations.** Donations make sense for libraries developers use directly; they don't make sense for B2B SaaS where customers will gladly pay $19–$249/mo for software that increases their margin. Have GitHub Sponsors enabled — treat any inbound as a bonus, never the plan.
- **Distribution > code.** The thing that scales is not the codebase. It's SEO content, the free tools, the accountant partner channel, and word of mouth from sellers who recovered money via the auditor (IDEAS #18). Build accordingly.

## Licensing strategy

### The honest framing

A pure open-source license (MIT / Apache 2.0) on the SaaS app means: AWS, a competitor, or a Russian outfit can spin up `ebay-soft-clone.com` next week using your code. For an infrastructure tool that's healthy (someone else's hosting bill doesn't kill you). For a small B2B SaaS founded by one person, it's product suicide.

A proprietary, closed license means: customers can't audit how their financial data is handled, can't fork if you disappear, and have to trust your word on security.

The middle path is **source-available** — pioneered by Sentry (BSL), MongoDB (SSPL), Elastic (ELv2), Couchbase (BSL). Code is on GitHub; anyone can read, modify, run for non-production internal use; nobody can offer it as a competing hosted service. After N years it converts to a real OSS license, so the long-tail community guarantee exists.

### Recommended licenses

| Repo / artifact                                         | License                                    | Why                                                                 |
|---------------------------------------------------------|--------------------------------------------|----------------------------------------------------------------------|
| **`ebay-soft/ebay-soft`** (the SaaS — services, web, infra) | **BSL 1.1** with `change_date` = `today + 4y`, `change_license` = `Apache-2.0`, additional grant: "permitted use does not include offering this software as a hosted commercial service to third parties" | Customers can audit; competitors can't run a clone; in 4 years it becomes real OSS — a credible long-term commitment |
| **`ebay-soft/ebay-sdk-java`** (typed Java client for eBay's Sell APIs) | **Apache 2.0**                  | Building this in the open compounds the brand; useful far beyond ebay-soft |
| **`ebay-soft/ebay-sdk-ts`** (TS client for browser/Node)         | **Apache 2.0**                  | Same                                                                 |
| **`ebay-soft/listing-scorer`** (title-scoring algorithm + corpus) | **Apache 2.0**                  | Public good; we benefit from PRs                                     |
| **`ebay-soft/free-tools`** (fee calculator, image bg-remover, etc.) | **Apache 2.0**                | These are also our marketing site; making them open invites links and inbound |
| **`ebay-soft/docs`** (engineering docs from `docs/`)           | **CC BY 4.0**                   | Knowledge sharing                                                    |
| **`ebay-soft/specs`** (our published OpenAPI specs once the API is stable) | **Apache 2.0**             | So clients can be generated by anyone                                |

### What BSL 1.1 looks like in practice

The repo root has a `LICENSE` file with the BSL text, a parameter block:

```
Licensor:             ebay-soft sp. z o.o. (or your entity)
Licensed Work:        ebay-soft, version X.Y.Z
Additional Use Grant: You may make use of the Licensed Work, provided that you may not offer the
                      Licensed Work to third parties on a hosted or embedded basis in order to
                      compete with the Licensor's paid version of the Licensed Work.
Change Date:          Four years after the date the Licensed Work is published.
Change License:       Apache License, Version 2.0
```

A `LICENSING.md` explains it in human English at the top of the README. The "change date" auto-rolls per release — each release's clock starts when that release ships, so old code becomes Apache 2.0 four years after it was published.

### What we will NOT do

- **AGPL** — too restrictive for our customers' downstream tooling, scares enterprises.
- **SSPL** (MongoDB-style) — not an OSI-approved license, alienates the community.
- **Custom homegrown license** — nobody trusts these; use BSL.
- **Pure proprietary with no source** — gives up the trust-and-audit benefit for no upside.

## Donations vs subscriptions

Be honest with yourself about which model fits.

**Donations work when:**

- The user is technical and altruistic.
- The tool is a library/CLI/framework with no per-user infrastructure cost.
- The marginal user costs you nothing.

**Donations don't work when:**

- The user is a small business owner whose mental model is "tools = expense line, software pays for itself or it goes."
- The tool runs in your cloud (every user adds infra cost).
- You need ~$85k/yr to live and ~$12k/yr to run servers.

eBay sellers are the second category. They will pay $49/mo if it earns them $500/mo. They will not donate $5/mo to "support the project."

That said, **GitHub Sponsors on the org is free to set up**, and a small number of friendly developers or accounting partners may sponsor symbolically. Enable it. Don't pitch it. Don't put it on the marketing site.

A `FUNDING.yml` in each public repo:

```yaml
github: [ebay-soft]
custom: ["https://ebay-soft.com/pricing"]
```

— points sponsors at the actual revenue channel.

## Going from solo to small team

A realistic staffing curve as MRR grows:

| MRR         | Headcount                                              | Notes                                                  |
|-------------|--------------------------------------------------------|--------------------------------------------------------|
| $0          | 1 founder, full-time                                   | The plan in [ROADMAP.md](ROADMAP.md)                  |
| $5k         | 1 founder + a part-time designer + part-time copywriter| Already enough to ship the MVP and the marketing site  |
| $15k        | + first full-time engineer                             | Hire someone strong on the WebFlux/Postgres side       |
| $30k        | + customer-success / support engineer                  | Critical at this stage — support quality is the moat   |
| $60k        | + second engineer (Python/ML), + part-time accountant  | The ML features need real focus                        |
| $100k       | + sales / partnerships (accountant channel)            | Channel revenue unlocks                                |
| $200k+      | Real exec team                                         | If we get here, congratulations                        |

The order matters. Don't hire a second engineer before a support person — engineers without time-on-tickets stop building the right things.

### Where to find the first hires

- **Indie Hackers, eBay seller community Slack/Discord** — for support hires; someone who's sold on eBay themselves is gold.
- **JoinEU, Honeypot, OTTA** — for European engineers.
- **GitHub network from your open-source artifacts** — best engineering pipeline; this is the *real* reason to open-source the SDKs.

## Contribution model (for the open repos)

For the Apache-2.0 repos (`ebay-sdk-java`, `ebay-sdk-ts`, `listing-scorer`, `free-tools`):

- **CLA** required (Contributor License Agreement). Use [CLA Assistant](https://github.com/cla-assistant/cla-assistant) — free, GitHub-integrated. Required so we can later relicense or change strategy without chasing every contributor for sign-off.
- **DCO** (Developer Certificate of Origin) is the lighter-weight alternative if a CLA feels heavy. Either is fine.
- **`CONTRIBUTING.md`** in every public repo: how to set up, how to run tests, how PRs are reviewed, code of conduct (Contributor Covenant).
- **`CODE_OF_CONDUCT.md`** — Contributor Covenant 2.1.
- **`SECURITY.md`** — vulnerability disclosure email + PGP key; 90-day timeline. Same content as `docs/SECURITY.md`'s disclosure section.
- **Issue templates** — bug, feature, question.
- **Auto-label and auto-assign** with GitHub Actions to keep triage cost near zero.

For the source-available main repo (BSL): same files, but contributions explicitly assigned to the licensor via CLA. We don't accept anonymous PRs that complicate future licensing.

## Trademark

Register the wordmark **"ebay-soft"** in the EU (EUIPO) and US (USPTO) early — even if just the wordmark, not the logo (cheaper). Approx €850 EU + $250–$350 US. Without this, anyone can fork the code and call their fork "ebay-soft" too; the license can't stop trademark abuse, but a registered trademark can.

Note: **"eBay" is a trademark of eBay Inc.** We are not affiliated and must not imply we are. The brand "ebay-soft" describes "software for eBay sellers" — defensible as descriptive use, but get a real trademark lawyer's read before spending big on the brand. Worst-case fallback name: pick something that doesn't include "ebay" (e.g. "Margin Hub", "Sellerline"). Don't bet the whole brand on a name that eBay could challenge.

## Sustainability metrics

A healthy open-core / source-available project tracks:

| Metric                                            | Why it matters                                       |
|---------------------------------------------------|------------------------------------------------------|
| Stars on `ebay-sdk-java` / `ebay-sdk-ts`          | Top-of-funnel for engineering hires + indirect SEO   |
| External contributors / quarter                   | Health of the open repos                              |
| Public-issue-to-close median time                 | Trust signal; closing fast = engaged maintainers     |
| Free-tool monthly uniques                         | Marketing funnel input                                |
| Sign-up conversion from free-tool page            | Whether the open artifacts pay for themselves        |
| BSL-repo views / clones                           | Customer trust signal — auditors actually look       |
| GitHub Sponsors revenue                           | Vanity metric; never optimize for it                  |

If `external contributors / quarter` is zero for a year, the open repos are doing their marketing job but not their community job. That's still fine; just be honest about it.
